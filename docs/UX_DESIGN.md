> Version: 1.0.0
> Last Updated: 2026-05-07
> Status: 设计完成，待实施
>
> 实现选项 C（抽取可视化 + 用户确认）+ 双输入模式（自然语言/语音 vs 勾选清单）+ 新手引导。

---

## 0. 设计目标

| # | 目标 | 当前 MVP 状态 |
|---|---|---|
| 1 | 用户能用最自然的方式描述症状（说话、打字、点选）| ❌ 仅 textarea |
| 2 | 系统怎么理解用户话**永远可见可改** | ❌ 完全黑盒 |
| 3 | 信息缺失时不偷偷兜底，让用户补充 | ❌ 当前走 R999 兜底 |
| 4 | 第一次使用知道是干什么的、隐私边界 | ❌ 无引导 |

```

│  ← 返回   确认信息      ?  历史          │
├────────────────────────────────────────┤
│                                          │

│                                          │

│                                          │
│  📌 距离上次化疗：3 天 ✓ [可改]           │
│                                          │
│  ┌────────────────────────────────────┐ │
│  │ 🌡️ 发热                       [✕] │ │ <- ✕ 删除该症状
│  │  ⚠️ 体温：[___] °C  *必填          │ │ <- 红色高亮缺失
│  │  持续时间：[__] 小时   选填          │ │
│  └────────────────────────────────────┘ │
│                                          │
│  ┌────────────────────────────────────┐ │
│  │ 😖 恶心                       [✕] │ │
│  │  严重程度：(•) 轻 ( ) 中 ( ) 重     │ │ <- LLM 推断 mild
│  │  ☑ 影响进食/日常活动                │ │
│  └────────────────────────────────────┘ │
│                                          │
│  ┌─ + 补充其他症状 ──────────────────┐  │ <- 让用户补漏
│  │ 您还有以下症状吗？                  │  │
│  │ ☐ 呕吐  ☐ 腹泻  ☐ 疲劳  ☐ 皮疹  │  │
│  │ [展开全部 12 类]                    │  │
│  └────────────────────────────────────┘  │
│                                          │
│ ┌──────────────────┬────────────────┐  │
│ │ ← 重新描述        │  确认 看结果 → │  │ <- 缺必填时 disabled
│ └──────────────────┴────────────────┘  │
└────────────────────────────────────────┘
```

#### 关键交互规则

1. **顶部 incomplete banner**：仅 `completeness.is_complete=false` 时显示，红/黄背景
2. **必填字段**：必填且为空时红色边框 + 红色 `*必填`，"确认"按钮 disabled
3. **可选字段**：留空允许，"选填" 灰色提示
4. **删除症状 ✕**：点击立即从列表移除（不弹确认 — MVP 简单为先）
5. **补充症状区**：自然语言 mode 下展示，让用户补 LLM 漏抽的；form mode 下隐藏（已经选过了）
6. **"重新描述"**：返回 InputPage 保留原文（不清空 textarea）

#### 数据流

```
用户提交 InputPage
  ↓
POST /api/v1/assessments/extract { user_id, raw_input_text, input_source }
  ↓
后端 LLM 抽取 + CompletenessChecker（不写库）
  ↓
返回 { parsed_symptoms, completeness }
  ↓
前端进入 ConfirmExtractionPage，状态保存在 React state（不持久化）
  ↓
用户编辑/确认
  ↓
POST /api/v1/assessments { user_id, raw_input_text, parsed_symptoms (用户确认版), idempotency_key }
  ↓
后端走规则引擎 + 写库
  ↓
ResultPage 显示
```

**注意**：用户确认的 ParsedSymptoms 写入 `assessment.parsed_symptoms` JSONB；
**`extraction_source` 字段需要新加值 `"user_confirmed"`** 表示这是经用户审阅过的，不是纯 LLM 抽取。

## 3. 后端 API 改造（Option III: 无状态预览 + 现有接口接收 confirmed）

### 3.1 新增 `POST /api/v1/assessments/extract`

**用途**：**仅抽取**，不写库。给前端做预览/确认用。

```yaml
POST /api/v1/assessments/extract
Request:
  {
    "user_id": "uuid",
    "input_source": "free_text" | "checklist",
    "raw_input_text": "..."     # free_text 时必填
    "form_payload": {...}        # checklist 时必填，含 symptoms[] + context
  }

Response 200:
  {
    "parsed_symptoms": {
      "symptoms": [...],
      "context": {...},
      "confidence": 0.78
    },
    "completeness": {
      "is_complete": false,
      "missing_slots": [
        {"symptom_id": "fever", "missing_fields": ["numeric_value"]}
      ]
    },
    "extraction_model_version": "deepseek-v4-flash"
  }

Response 422 (LLM 失败):
  {
    "reason": "extraction_failed" | "low_confidence",
    "message": "无法解析您的描述，建议改用清单模式"
  }
```

**实现要点**：
- 服务端不持久化任何东西（无 temp_id）
- LLM 失败时建议前端切换到 form mode
- form mode 输入直接构造 ParsedSymptoms 跳过 LLM（form_payload 已结构化）

### 3.2 修改现有 `POST /api/v1/assessments`

入参新增 `parsed_symptoms` 字段（用户确认版的结构化症状）：

```yaml
POST /api/v1/assessments
Request:
  {
    "user_id": "uuid",
    "session_id": "...",
    "input_source": "free_text" | "checklist",
    "idempotency_key": "uuid",
    "raw_input_text": "...",       # 总是保留原文
    "parsed_symptoms": {            # 必填：用户确认/编辑后的最终版
      "symptoms": [...],
      "context": {...},
      "confidence": 0.78
    }
  }

Response: 同现有 AssessmentResult
```

**实现要点**：
- Orchestrator **跳过 LLM 抽取**（用入参的 parsed_symptoms 直接进规则引擎）
- `assessment.parsed_symptoms` JSONB 存用户确认版
- `symptom_observation.extraction_source = 'user_confirmed'`
- `assessment.extraction_confidence` 仍为 LLM 给的值（即使被用户编辑），便于审计追溯

### 3.3 兼容性

旧的 `POST /assessments` 调用（不传 `parsed_symptoms`）应仍然工作：
- 服务端检测无 `parsed_symptoms` → 降级到原 Orchestrator 流程（自动调 LLM）
- 给"不想走两步"的 API consumer 留口子（如未来移动 app、CLI、其他系统集成）

## 4. 数据模型小补充

### 4.1 新增 extraction_source 值

```sql
-- 现有
ALTER TABLE symptom_observation
  ALTER COLUMN extraction_source TYPE VARCHAR(20);  -- 从 16 扩到 20

-- 允许值扩展
-- 'llm' (现有)        : 纯 LLM 抽取，未经用户编辑
-- 'checklist' (现有)  : 纯表单输入
-- 'user_confirmed'    : LLM 抽取 + 用户审阅/编辑（新增）
-- 'user_added'        : 用户在 confirm 页主动补充的症状
```

### 4.2 case_review 触发条件扩展

新增触发源：
- `user_edited_extraction`：用户在 confirm 页**修改了** LLM 抽取的结果（说明 LLM 抽错了）
- `user_added_symptom_in_confirm`：用户**补充了** LLM 漏抽的症状

这两种都是**高价值的 L3 学习信号** —— 比 `user_disagreement` (👎) 更具体，能直接指向 prompt 该改哪里。

### Phase v3 — 每日追踪（Apple Health 月经追踪式）

灵感来自 Apple Health：低门槛、高频率、可视化。让用户每天 30 秒打个卡，
积累成趋势数据。这本身就是 Advanced 时序规则（R030 等）的真实数据来源。

#### 主屏：本月日历

```
├─────────────────────────────────┤
│  日 一 二 三 四 五 六              │

│           1  2  3  4               │

```

#### 打卡入口：3 步快速记录（区别于完整评估）

```
Step 1: "今天有没有不舒服的地方？"
  [完全没有]  [有一点]  [比较难受]  [需要立即就医]

Step 2 (如选有): 多选今天出现的（最多 5 个）
  ☐ 发热  ☐ 恶心  ☐ 疲劳  ☐ 手脚麻
  ☐ 腹泻  ☐ 口腔溃疡  ☐ 其他

Step 3 (如有具体症状): 严重程度 1-5 滑块
  发热程度: ●─────  (温和)
  恶心程度: ───●──  (中等)

[完成打卡 ✓]
```

3 步走完 ≤30 秒。不强制走完整 confirm 流程。

#### 数据模型变化

新增 `daily_checkin` 表（轻量版 assessment）：

```sql
CREATE TABLE daily_checkin (
    id              UUID PRIMARY KEY,
    user_id         UUID NOT NULL,
    checkin_date    DATE NOT NULL,
    overall_status  VARCHAR(16),  -- none/mild/moderate/severe
    symptoms        JSONB,        -- 简化版，3 步收集的
    created_at      TIMESTAMPTZ NOT NULL,
    UNIQUE (user_id, checkin_date)  -- 每天最多一条
);
```

**与 assessment 的关系**：daily_checkin 是低门槛的"打卡"，
当 `overall_status='severe'` 或多症状时，自动跳转到完整 `assessment` 流程。

#### 推送 / 提醒

- 每天默认 18:00 提醒（用户可调）
- 连续打卡奖励：streak 显示 "您已连续 7 天记录"
- 漏打 3+ 天时温和提醒，但不强推

#### 技术选型

- 提醒：iOS push（PWA + Web Push API）/ 浏览器通知
- 日历组件：直接用 `react-day-picker` 或自写（MVP 自写避免依赖）
- 趋势驱动：每天打卡后自动喂给 Advanced 时序视图 `v_user_trend_7d`，
  如检测到趋势恶化（`R030_worsening_fatigue_trend`）主动通知

## 6. 几个关键 UX 决策

| 决策 | 选择 | 反对意见 | 为什么仍这样选 |
|---|---|---|---|
| Onboarding 是页面还是 Modal | **页面** | Modal 更现代 | 首次进入信息密度高，全屏更不被打扰 |
| 输入模式默认哪个 | **自然语言** | 表单更稳 | 大多数用户更习惯说话；表单是兜底 |
| confirm 页是否强制 | **强制（自然语言模式）** | 多一步累 | 信任建立的核心环节，不能跳 |
| confirm 页是否强制（form 模式）| **也强制** | form 已经结构化了 | 一致性 + 给改字段的入口 |
| 缺必填时"确认"按钮 | **disabled** | 应该让用户提交看个粗略结果 | 医疗场景需要精确；缺关键信息走 R999 兜底是反 UX |
| 语音识别失败怎么办 | **静默 + 提示** | 自动重试 | 用户应控制何时重新录 |
| 删除症状 | **直接删，不确认** | 误触怎么办 | confirm 页有撤销/重选的余地 |
| 隐私同意 | **必须勾选才能继续** | 用户嫌烦 | GDPR / PIPL 合规要求 |

## 7. 视觉设计原则（为后续 polish 阶段记下）

- **色彩**：风险等级 high=红、medium=橙、low=绿；缺失字段红边框
- **字号**：医疗用户老年人多，正文 ≥16px
- **触控目标**：按钮 ≥44pt（iOS HIG 标准）
- **状态反馈**：每次提交立即显示 loading（"正在分析您的描述..."），不留空窗
- **错误信息**：永远人话化，不要 "HTTP 422" 这种

---

## 8. 风险点

1. **LLM 抽取慢**：5-15 秒，需 loading 状态，可考虑 streaming
2. **首次用户跳过 onboarding**：要在 InputPage 也保留入门提示
3. **form mode 占屏多**：12 类全展开 ~3 屏，要折叠默认 / 智能默认
4. **confirm 页太复杂**：用户可能直接放弃。需 A/B 测下"完整 confirm" vs "最简版"
